# Database Architecture

The database layer is PostgreSQL, managed through SQLAlchemy ORM models and Alembic migrations. It stores canonical article feed rows plus enrichment records for extracted article text, AI summaries, entities, countries, and funding values.

## Runtime Database

Docker Compose defines the `postgres` service:

```yaml
postgres:
  image: postgres:17
  environment:
    POSTGRES_DB: startup_news
    POSTGRES_USER: startup_news
    POSTGRES_PASSWORD: startup_news
  ports:
    - "5432:5432"
  volumes:
    - postgres_data:/var/lib/postgresql/data
```

The named volume `postgres_data` persists local database state across container restarts.

## Connection URLs

Local host connection from `.env.example`:

```env
DATABASE_URL=postgresql+asyncpg://startup_news:startup_news@localhost:5432/startup_news
```

Container connection from `docker-compose.yml`:

```env
DATABASE_URL=postgresql+asyncpg://startup_news:startup_news@postgres:5432/startup_news
```

The difference is the host:

- `localhost` when a local process connects through the published port.
- `postgres` when a container connects through the Compose network.

## Migration Ownership

Migrations live in:

```text
services/backend/alembic
```

The initial migration is:

```text
services/backend/alembic/versions/20260608_0001_create_articles.py
```

Docker Compose runs migrations with the `migrate` service before starting the API and ingester:

```bash
alembic upgrade head
```

## Current Schema

### `articles`

| Column | Type | Nullable | Notes |
| --- | --- | --- | --- |
| `id` | UUID | No | Primary key, generated by application default. |
| `source` | varchar(255) | No | Source plugin identifier, currently `google_news`. |
| `title` | varchar(1024) | No | Article title from the feed entry. |
| `url` | varchar(2048) | No | Canonical source URL. Unique. |
| `external_id` | varchar(512) | Yes | Source-specific feed id or guid when available. |
| `source_url` | varchar(2048) | Yes | Feed/listing URL or pre-normalized source URL for traceability. |
| `published_at` | timestamptz | Yes | Published or updated timestamp from the feed when available. |
| `content` | text | Yes | RSS summary or content HTML/text when available. |
| `created_at` | timestamptz | No | Insert timestamp, defaults to database `now()`. |

Constraints:

- Primary key on `id`.
- Unique constraint `uq_articles_url` on `url`.
- Index `ix_articles_source` on `source`.
- Index `ix_articles_published_at` on `published_at`.

## Write Pattern

The ingestion service writes articles in batches:

1. Fetch raw articles from all registered source plugins.
2. Deduplicate fetched articles by URL in memory.
3. Query the database for URLs that already exist.
4. Insert only new article rows.
5. Commit the transaction.
6. Roll back and return zero created rows if a unique constraint race occurs.

The unique URL constraint is the final line of defense against duplicate records.

## Read Pattern

The API reads from the `articles` table and orders records by:

1. `published_at DESC`
2. `created_at DESC`

The dashboard requests:

```text
GET /articles?limit=20&offset=0
GET /articles/facets
```

Search, source, entity, funding range, and country filters are applied by the backend so filtering works across the full dataset, not only the currently loaded page.

## Indexing

The schema defines primary key, unique URL, source, and published timestamp indexes. These support duplicate checks, source filters, and latest-article ordering.

Enrichment indexes cover enrichment status, startup country, publisher country, normalized USD funding amount, and typed entity name filters.

A future `created_at` index can be added if operational queries frequently sort or filter by ingestion time.

## Data Retention

There is no automated retention policy in Phase 1. Articles remain in the database until manually deleted or until the local Docker volume is removed.

To reset local data:

```bash
docker compose down -v
```

This removes the `postgres_data` volume and all stored articles.

## Future Schema Ideas

Useful next tables as the product grows:

- `sources`: source metadata, health, enabled status, polling interval.
- `ingestion_runs`: started time, finished time, fetched count, created count, error details.
- `funding_rounds`: normalized amount, currency, round type, company, and investor fields.
- `article_snapshots`: raw source payloads for traceability and reprocessing.

These should be introduced only when the application needs them. The current single-table design keeps Phase 1 easy to reason about.
